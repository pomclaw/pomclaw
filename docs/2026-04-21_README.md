## ✅ 核心改动 5 句话总结

### 1️⃣ 消息层 - InboundMessage 添加 agent_id
从无到有地为消息传递 agent_id，支持端到端追踪
```go
type InboundMessage struct {
    AgentID string  // ← 新增
    // ...
}
```

### 2️⃣ Store 管理 - 多 Agent 缓存隔离
创建 StoreManager 支持 1000+ agent 并发缓存
```go
storeManager := NewStoreManager(cfg, db)
sessionStore, _ := storeManager.GetSessionStore(agentID)  // 隔离
```

### 3️⃣ AgentLoop 改造 - 运行时 agent_id 确定
从启动时固定的 agent_id，改为运行时从消息动态获取
```go
sessionStore, _ := al.storeManager.GetSessionStore(opts.AgentID)
stateStore, _ := al.storeManager.GetStateStore(opts.AgentID)
```

### 4️⃣ 工具系统 - agent_id 上下文传递
工具执行时传递 agent_id，实现工具级隔离
```go
tools.WithToolContext(ctx, channel, chatID, agentID)
```

### 5️⃣ HTTP 通道 - 多租户请求接口
实现 HTTP 服务，每个请求必须提供 agent_id
```go
POST /api/v1/chat
{
  "agent_id": "tenant-001",
  "session_key": "s1",
  "message": "Hello"
}
```

## ⚠️ 最重要的 3 点

### 1️⃣ SessionKey 隔离
- 确保不同 agent 的 session 不冲突
- SessionKey 由客户端生成，数据库中通过 agent_id 隔离
- 建议 session key 格式：`{tenant_id}:{unique_id}`

### 2️⃣ 缓存大小限制
- 防止 5000+ agent 导致内存爆炸
- 使用 LRU 策略，最多缓存 5000 个 agent 的 store
- 监控内存占用，告警阈值：500MB 警告，1GB 重告

### 3️⃣ 数据污染监控
- 持续监控跨租户数据泄露
- SQL 查询检查：某个 session 是否有多个 agent_id
- 严重问题立即告警和回滚

## 💬 常见问题

### Q: 需要多长时间完成改造？
**A:** 7.5 天（1 个工程师，包括测试）。如果并行 2-3 人，可压到 3-4 天。

### Q: 改造后性能会变差吗？
**A:** 不会。多了 Store 查询和缓存操作，但使用 LRU 和多级缓存后，整体性能会更好。

### Q: 现有数据会迁移吗？
**A:** 不需要迁移。新版本启动后，现有 agent 默认使用 "default" agent_id，无缝兼容。

### Q: 可以只改部分 Agent 吗？
**A:** 可以。使用 feature flag 控制，旧 agent 用旧逻辑，新 agent 用新逻辑。

### Q: 多租户间真的完全隔离吗？
**A:** 是的。数据库层有 PRIMARY KEY (agent_id, session_key) 保证，缓存层也按 agent_id 隔离。
